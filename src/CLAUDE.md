# CLAUDE.md — Orchestration Flow

이 문서는 작업 흐름과 의사결정 절차를 정의한다. 응답 포맷·도구 사용 규칙은 `AGENTS.md`, 정책 디테일은 `rules/*.md`를 참조한다.

## 시작 전 필수 절차

- **현재 상태 파악**: `git status`로 기존 변경사항 확인
- **파일 구조 이해**: 관련 코드와 의존성 파악
- **범위 준수**: 요청 범위를 벗어난 변경 금지
- **간단한 요청**: 즉시 실행, 복잡한 요청만 계획 단계 거침

## 사고·응답 언어 정책

- **사고(thinking) 단계**: 영어 — 더 정확한 reasoning
- **응답(output)**: 한국어 — 사용자 가독성 우선
- **코드/명령어/기술 용어/에러 메시지**: 원문(영어) 유지

## 워크플로우 오케스트레이션

### 1. 플랜 모드 기본값
- 3단계 이상 또는 아키텍처 결정 → 반드시 플랜 모드 진입
- 의도대로 흘러가지 않으면 즉시 STOP하고 재계획
- 모호함을 줄이기 위해 사전에 상세 스펙(질의 문답)으로 작성

### 2. 서브에이전트 전략
- 메인 컨텍스트 보호를 위해 적극 활용
- 리서치·탐색·병렬 분석은 서브에이전트에 위임
- 서브에이전트당 하나의 작업 (가이드 권장: low effort + 명시적 체크리스트)

### 3. 자기개선 루프
- 사용자 수정 받을 때마다 `MEMORY.md`에 패턴 기록
- 동일한 실수를 막는 규칙 작성 → 이후 즉시 반영
- 세션 시작 시 관련 레슨 검토

### 4. 완료 전 검증
- 작동 증명 없이 완료 표시 금지
- "시니어 엔지니어가 승인할까?" 자문
- 테스트 실행, 로그 확인, 정확성 입증

### 5. 우아함 추구 (균형)
- 명확하지 않은 변경: "더 우아한 방법이 없을까?" 자문
- 임시방편 같으면: "지금 알고 있는 모든 것을 고려해 명백한 솔루션 구현"
- 간단한 수정은 이 과정 생략 — 과잉 설계 금지
- **판단 기준**: "3개월 후 내가 이 코드를 이해할 수 있을까?"
- **리팩토링 신호**: 같은 패턴 3회 반복 (Rule of Three)

### 6. 자율 버그 수정
- 버그 리포트는 그냥 고쳐라 — 일일이 지시 구하지 말 것
- 로그·오류·실패한 테스트 직접 찾아 해결
- **근본 원인 우선**: 증상만 고치지 말 것
- **재발 방지**: 동일 유형 문제가 다른 곳에 없는지 확인

### 7. 병렬 실행 원칙
- 독립적 작업은 항상 병렬로 (단일 메시지에서 다중 Task 호출)
- 의존성 없는 3개 분석을 순차 실행하지 않는다

### 8. Git 및 변경 안전성 (CRITICAL)
- **금지**: 사용자가 명시적으로 요청하지 않은 `commit`, `push`, 브랜치 전략 변경
- **기존 변경 보호**: 사용자 변경을 임의로 되돌리지 않음
- **외부 변경 감지**: 예상치 못한 변경 발견 시 즉시 중단하고 확인
- **파괴적 명령**: `reset --hard`, `rm -rf`, `push --force`는 명시적 승인 필수

## Effort × 모델 정책

Anthropic 가이드(Opus 4.7) 기준 — 디테일은 `rules/performance.md` 참조.

| Effort | 모델 | 용도 |
|---|---|---|
| `low` | `claude-haiku-4-5` | 단일 도구 체크리스트, 명확한 범위 (서브에이전트, 분류, 빠른 조회) |
| `medium` | `claude-sonnet-4-6` | 균형형 — 도구 호출 + 일부 추론 |
| `high` | `claude-sonnet-4-6` | 복잡 추론·신중한 판단 |
| `xhigh` | `claude-opus-4-7` | 코딩·탐색·다단계 (반복 도구 호출, 깊은 검색) |
| `max` | — | 진정한 최전선만 — 일반 워크로드에는 비권장 |

**핵심 원칙**: "프롬프트 주변을 길게 쓰기보다 effort를 올려라." Opus 4.7은 더 낮은 effort에서 요청된 것만 한다.

## 에이전트 자동 호출 시점

- 코드 작성 직후 → `code-reviewer`
- 새 기능/버그 수정 → `tdd-guide` (테스트 우선)
- 빌드 실패 → `build-error-resolver`
- 커밋 전 → `security-reviewer`

## 완료 보고 형식

1. **변경 사항**: 무엇을 바꿨는지 (파일 경로:라인)
2. **변경 이유**: 왜 바꿨는지
3. **검증 결과**: 어떻게 검증했는지 (테스트·빌드 결과)
4. **다음 단계**: 자연스러운 후속 작업이 있으면 번호 목록으로

**예시:**
```
변경: src/auth/login.ts:42-58 로그인 검증 로직 수정
이유: 빈 이메일 입력 시 서버 에러 → 클라이언트 검증 추가
검증:
  - 단위 테스트 통과 (npm test auth.test.ts)
  - E2E 테스트 통과 (정상 + 빈 입력 케이스)
다음 단계:
  1. 비밀번호 검증에도 동일 패턴 적용 검토
  2. 에러 메시지 다국어 지원
```

## 핵심 코드 품질 원칙

- **단순함 우선**: 최소 변경. YAGNI / KISS.
- **게으름 금지**: 근본 원인 분석. 임시 수정 없음. 시니어 기준.
- **최소 영향**: 필요한 부분만. 사이드 이펙트 생성 금지.
- **두 가지 대안 이상 검토** → 트레이드오프 명시 → 되돌릴 수 있는지 확인.
