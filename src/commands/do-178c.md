# Do-178c Command

## What This Command Does

Loads the `do-178c` skill and applies its assurance methodology to the target named in `$ARGUMENTS` (a path or a task description). For that target it:

- **(a) Assigns or uses an assurance tier (A-E)** based on the worst-case blast radius of the change.
- **(b) Reports which gates that tier dials** — coverage depth from `tdd-workflow`, verification breadth from `verification-loop`, review independence from `code-reviewer` and `assurance-auditor`, and `security-review` sign-off.
- **(c) Checks bidirectional traceability and structural-coverage expectations** for the tier — every requirement maps to a test and back, and no requirement or code path is orphaned.

Honors an optional `--tier A|B|C|D|E` override when you already know the criticality and want to skip classification.

## When to Use

- Starting a safety-critical or high-blast-radius change (auth, payments, crypto, DB migration) and you need to fix the rigor up front.
- Deciding how much rigor a change actually needs, instead of over- or under-testing by reflex.
- Auditing whether a finished change met its tier's bar before you call it done.

## How It Works

1. **Classify the tier** by worst-case blast radius — what breaks, how reversible it is, and what it touches (data, money, security, persistent state). Use `--tier` if supplied; otherwise infer from the target. This is the one net-new decision the command owns.
2. **List the dialed gates** for that tier: the coverage target from `tdd-workflow`, the verification depth from `verification-loop`, whether review must be independent (`code-reviewer`, plus `assurance-auditor` for the top tiers), and whether `security-review` sign-off is mandatory.
3. **Check traceability and coverage**: confirm bidirectional traceability (each requirement has a covering test, each test traces to a requirement) and that structural coverage meets the tier expectation — statement then decision/branch. MC/DC is the A-tier ideal but is not gated, since most JS/TS/Python stacks cannot measure true MC/DC.
4. **For A-tier, hand off to `assurance-auditor`** for the independent audit, since the author cannot self-certify catastrophic-impact work.

## Example Usage

```
User: /do-178c src/payments

Agent: Loaded `do-178c`. Classified src/payments as Tier A (Catastrophic) —
handles money movement and persistent ledger state; failures are irreversible.

Dialed gates for Tier A:
  - tdd-workflow: requirements-based tests + decision/branch coverage
    (MC/DC aspirational, not gated)
  - verification-loop: full (build, type, lint, test, security)
  - code-reviewer + assurance-auditor + human review (independent)
  - security-review: sign-off required

Traceability + coverage check:
  - REQ "refunds must be idempotent" has NO covering test — untested
    requirement; add a requirements-based test before merge.
  - reconcileLedger() in processor.ts:87 traces to no requirement —
    orphan function; either link it to a requirement or remove it.

Next: handing off to `assurance-auditor` for the independent A-tier audit.
```

## Related Commands

- `/tdd` — drive the requirements-based tests and coverage the tier demands.
- `/verify` — run the build/type/lint/test verification depth for the tier.
- `/code-review` — independent quality and security review of the diff.
- `/deps` — confirm dependency direction and coupling for the changed boundary.
- `/security-review` — required sign-off for A-tier (auth, payments, secrets, input).

Full methodology lives in the `do-178c` skill — follow that as the source of truth.
