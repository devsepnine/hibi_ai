# Abstraction and Module Boundaries

모듈은 복잡성을 통제할 때 사고의 단위가 되는 대상이다. 모듈의 경계를 잘 긋는 일은 의존성 설계에서 가장 영향력이 큰 결정인데, 경계가 곧 나머지 시스템이 무엇을 볼 수 있고, 무엇에 의존하며, 무엇 때문에 깨질 수 있는지를 결정하기 때문이다. 이 문서는 개념적 판단을 위한 자료다. 기계적으로 따르는 규칙 목록이 아니라, 모듈이 무엇을 노출하고 왜 그렇게 하는지를 어떻게 사고할지를 다룬다.

## 모듈은 세 종류의 지식이다

모든 모듈은 자신이 다루는 도메인 지식을 세 가지 통(bucket)으로 나눈다.

- **Interface** — 다른 요소가 상호작용할 수 있도록 *공개한* 지식이다. 즉 계약이다. 타입, 시그니처, 이름, 보장(guarantee)이 여기 속한다. 한 번 공개하면 다른 요소들이 그것에 의존하게 된다.
- **Implementation** — 실제로 *은닉한* 지식이다. 진짜 로직, 데이터 배치, 알고리즘. 바깥에서는 이것의 존재조차 알 필요가 없다.
- **Context** — 직접 *구현하지 않고* 외부에 가정하거나 요구하는 지식이다. 환경, 사전 조건, 주입받는 협력자, 암묵적 가정이 여기 속한다.

```
        ┌─────────────────────────────┐
        │            Module           │
        │                             │
  ──────┤  Interface  (published)     │   호출자가 보고 의존하는 것
        │ ─────────────────────────── │
        │  Implementation (hidden)    │   실제 작업, 바깥에서 보이지 않음
        │ ─────────────────────────── │
        │  Context    (unimplemented) │   외부 세계에 가정하는 것
        └──────────────▲──────────────┘
                       │
                  환경이 공급함
```

모듈을 설계한다는 것은 어떤 지식을 어느 통에 넣을지를 결정하는 행위다. 같은 사실이라도 공개할 수도, 은닉할 수도, 외부에 요구할 수도 있으며 그 선택에는 대가가 따른다.

### Interface는 조용히 회수할 수 없는 약속이다

Interface에 담긴 것은 모두 호출자가 결합하게 될 대상이다. 공개의 비용은 시그니처를 작성하는 행위 자체가 아니다. 그 시그니처에 대한 이후의 모든 변경이 바깥으로 파급된다는 점이다. 따라서 각 지식 조각에 대해 던질 질문은 "이걸 노출하면 유용한가?"가 아니라 "이 모듈이 살아 있는 동안 내가 여기에 묶여도 괜찮은가?"이다.

### Implementation은 자유가 보존되는 곳이다

Implementation에 담아 둔 지식은 아무도 그것에 의존하지 않으므로 누구에게도 알리지 않고 바꿀 수 있다. 더 많은 것을 implementation으로 밀어 넣을수록 모듈은 더 자유롭게 진화한다. 은닉은 은닉 자체가 목적이 아니라, 리팩터링 능력을 보존하는 메커니즘이다.

### Context는 빌려 온 복잡성이다

Context는 모듈이 소유하기를 거부하고 주변에 요구하는 지식이다. 일부 context는 불가피하다(모듈은 작용할 *무언가*가 필요하다). 그러나 환경에 부과하는 모든 요구는 호출자가 충족해야 할 사전 조건이며, 조용히 거짓이 될 수 있는 가정이다. Context는 가장 조용한 결합의 원천이다. 시그니처에는 드러나지 않지만 런타임에는 실재하기 때문이다.

## 지식 분류: 도메인 특화 vs 일반

어떤 모듈이든 그 안의 도메인 지식은 서로 얽힌 두 가닥의 혼합이다.

- **도메인 특화 지식(domain-specific)** — *이* 문제에 대해서만 참인 사실이다. "주문은 결제가 승인되기 전에는 출고될 수 없다." "마켓은 종가(close price)로 정산된다."
- **일반 지식(general)** — 도메인과 무관하게 참인 사실이다. 정렬, 재시도, 캐싱, 검증, 직렬화 같은 것들.

무엇을 공개할지 결정할 때, 당신은 사실상 어느 가닥을 노출할지를 결정하고 있는 것이다. 각 선택은 무언가를 얻고 무언가를 잃는다.

### 일반 지식을 공개하기

일반 가닥을 노출하면(예: 제네릭 `Repository<T>`, 제네릭 `retry(fn)`) **재활용성과 호환성**이 올라간다. Interface가 호출자를 특정 문제에 묶지 않으므로, 더 많은 호출자가 더 많은 도메인에 걸쳐 연결될 수 있다.

대가: 일반 interface는 *당신의* 도메인에 대해 거의 말해 주지 않으므로 호출자가 의미를 스스로 공급해야 하며, interface가 정보량이 적고 빈약한 형태로 흐를 수 있다.

### 도메인 특화 지식을 공개하기

특화 가닥을 노출하면(예: 도메인 형태의 명명된 연산을 가진 `MarketSettlementService`) **명확성과 신뢰성**이 올라간다. Interface가 무엇이 허용되고 무엇이 허용되지 않는지를 정확히 알려 주고, 잘못된 상태는 표현하기 어려워지며, 컴파일러와 이름이 의도를 담아낸다.

대가: 도메인 특화 interface는 그 도메인 안에서만 재활용 가능하다. 연결의 폭을 의미의 깊이와 맞바꾸는 것이다.

```typescript
// General — 연결성 높음, 도메인 의미 낮음
interface Repository<T> {
  find(id: string): Promise<T | null>
  save(entity: T): Promise<void>
}

// Domain-specific — 명확성 높음, 재활용 범위 좁음
interface SettlementLedger {
  recordSettlement(marketId: MarketId, price: ClosePrice): Promise<Receipt>
  // 절반만 만들어진 market을 "save"하는 것은 불가능하다. interface가 그것을 허용하지 않는다
}
```

어느 한쪽이 "더 좋은" 것은 아니다. 옳은 선택은 이 모듈이 재활용의 이음새 위에 있는지(일반을 선호), 아니면 절대 위반되어서는 안 되는 단단한 도메인 규칙을 담고 있는지(특화를 선호)에 달려 있다.

### 명확성과 신뢰성을 위해 context를 최소화하라

특화/일반 축과 무관하게, **context 통을 줄이는 것은 거의 항상 도움이 된다.** 모듈이 외부 세계에 대해 가정하는 바가 적을수록 고립된 상태로 이해하기 쉽고, 조용히 깨질 경로도 줄어든다. 세 개의 암묵적 전역(global)과 특정 호출 순서를 요구하는 모듈은 interface가 아무리 깔끔해 보여도 취약하다. 필요한 context는 *명시적이고 최소한으로* — 주입되고, 명명되며, 작업에 필요한 만큼만 작게 — 만드는 편이 좋다.

## 구체 지식이 아니라 추상화된 지식을 공개하라

이것이 세 통을 하나로 묶는 규칙이다. **어느 가닥을 공개하든, 구체적으로가 아니라 추상화하여 공개하라.**

Interface 안의 구체 지식 — 누출된 데이터 형태, 노출된 내부 enum, 현재 구현을 전제로 해야만 말이 되는 메서드 — 은 구현 디테일을 경계 너머로 끌고 간다. 시그니처가 기술적으로는 "interface"일 때조차, 너무 많이 누출되면 호출자가 결국 당신이 *무엇을* 약속하는가가 아니라 *어떻게* 동작하는가에 결합되고 만다. 이것이 구체 지식 누출이며, 하류의 모든 이에게 복잡성을 높인다.

반대로 추상화된 지식은 *how*는 비공개로 둔 채 *what*만 전달한다. 그리고 결정적으로, **추상화는 일관되어야 한다.** 한 메서드는 추상적이고 다음 메서드는 구체적인 interface는 균일하게 구체적인 것보다 더 나쁘다. 호출자가 자신이 어느 수준에서 작업하고 있는지 더 이상 예측할 수 없기 때문이다. 한 고도(altitude)를 정하고 경계 전체에 걸쳐 그것을 유지하라.

## 추상화의 세 가지 방법

추상화는 모호함이 아니다. 지식에는 *특수한* 부분과 *일반적인* 부분이 있고, 둘이 합쳐져 의미가 된다. 그러나 경계를 건너 전달되는 것은 일반적인 부분뿐이며, 특수한 부분은 뒤에 남는다. 상호작용하려면 지식을 건네야 하므로, 특수성과 일반성이 합쳐진 개념을 일반성을 앞세운 개념으로 재정의하게 된다. 그렇게 하는 데에는 서로 다른 측면에 작용하는 세 가지 방법이 있다.

### Modeling — 특정 관점으로 재해석

지식을 가져와 선택한 관점에서 다시 표현하고, 그 관점이 중요하게 여기는 바를 중심으로 재배치한다. 대상은 그대로 두되 부분들을 다시 놓아, 관련된 측면은 앞으로 나오고 무관한 측면은 뒤로 물러나게 한다.

> 은행 송금을 "출발지, 도착지, 금액을 가진 이벤트"로 *모델링*한다 — 같은 현실이지만, 감사 로그에 중요한 관점을 통해 재해석한 것이다.

Modeling은 **관점을 일반화**하여 추상화한다. 많은 구체 사례가 동일하게 보일 만큼 충분히 넓은 시점을 고른다.

### Categorization — 일반성을 향해 단순화

필요 없는 구분을 떨어내어, 남은 것이 더 단순하고 더 일반적인 개념이 되게 한다. 도달 범위를 얻기 위해 의도적으로 특수성을 버리는 것이다.

> `CreditCard`, `BankTransfer`, `Wallet`을 `PaymentMethod`로 *범주화*한다 — 차이는 버려지고 공유된 능력만 남는다.

Categorization은 **개념 자체를 일반화**하여 추상화한다. 속성은 적게, 적용 범위는 넓게.

### Grouping — 다른 존재론으로 편입

지식을 완전히 다른 준거 틀 아래로 끌어들인다 — *그것이 어떤 종류의 것인가*를 바꾼다. 가장 과감한 수다. 개념을 다른 존재론(ontology) 속에 새로 자리 잡게 하는 것이다.

> 보안 모델에서 "user", "service account", "API key"를 모두 `Principal`로 다룬다 — 이들은 서로 다른 종류의 엔티티지만, "권한을 부여받을 수 있는 무언가"만이 중요한 인가(authorization) 존재론 아래로 재편입된다.

Grouping은 **존재를 일반화**하여 추상화한다. 관점이나 속성 집합이 아니라 존재의 범주 자체를 바꾼다.

## 일반성은 연결성을 높인다

세 방법 모두 그 보상은 같다. **공개한 지식이 더 일반적일수록(그리고 더 적을수록) 연결성이 높아진다.** 충족해야 할 지식이 적을수록, 더 많은 것이 더 많은 맥락에서 더 적은 마찰로 그 경계에 연결될 수 있다.

그래서 각 방법은 자신의 고유한 방향으로 밀어붙일 때만 보상을 준다.

- Modeling은 **관점**을 일반화해야 한다.
- Categorization은 **개념**을 일반화해야 한다.
- Grouping은 **존재**를 일반화해야 한다.

디테일을 제거하지 않고 오히려 더하는 추상화, 또는 수준을 뒤섞는 추상화는 연결성을 낮추고 결합을 높인다 — 목표와 정반대다. 추상화는 경계를 이전보다 더 작고 더 일반적으로 남길 때에만 제값을 한다.

## 코드에서 이것이 드러나는 방식

이 문서의 판단은 규칙에서 구체화된다.

- **모듈이 무엇을 공개하고 무엇을 은닉할지 고르기** — interface를 좁게 유지하고 추상화를 일관되게 하는 메커니즘은 *Module boundaries and interface design* 규칙을 보라.
- **구현 디테일의 누출 막기** — *Concrete-knowledge leakage* 규칙을 보라.
- **Context를 최소화하고 명시적으로 만들기** — 의존성 방향과 주입된 context에 관한 규칙을 보라. 명명은 `coding-standards` skill이, 넓은 prop/parameter 표면 대신 composition으로 능력을 노출하는 방법은 `composition-patterns` skill이 다룬다.
- **일반과 도메인 특화 사이의 이음새가 어디에 있는지** — 일반적 영속성을 도메인 형태의 service 뒤에 두는 repository/service 계층화는 `backend-patterns` skill을 보라.
